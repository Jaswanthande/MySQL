## DAY 6 – TCL (TRANSACTION CONTROL LANGUAGE)

TRANSACTION MANAGEMENT IS THE PROCESS OF COMBINING ALL RELATED OPERA12:17 20-07-2026TIONS INTO A SINGLE UNIT OF WORK AND EXECUTING THEM ACCORDING TO THE RULE:

EITHER ALL OPERATIONS SHOULD SUCCEED OR NONE OF THEM SHOULD SUCCEED.

A TRANSACTION IS CONSIDERED A SINGLE UNIT OF WORK AND FOLLOWS THE PRINCIPLE:
EITHER ALL OR NONE.

EXAMPLE: FUND TRANSFER

CONSIDER A MONEY TRANSFER PROCESS BETWEEN TWO ACCOUNTS:

1. DEBIT THE AMOUNT FROM THE SENDER'S ACCOUNT.
2. CREDIT THE AMOUNT TO THE RECEIVER'S ACCOUNT.

BOTH OPERATIONS MUST BE EXECUTED AS A SINGLE TRANSACTION.
IF THE AMOUNT IS SUCCESSFULLY DEBITED FROM THE SENDER'S ACCOUNT BUT THE CREDIT TO THE RECEIVER'S ACCOUNT FAILS, IT WILL LEAD TO A DATA INCONSISTENCY PROBLEM.

TYPES OF TRANSACTIONS

1. LOCAL TRANSACTION

IN A LOCAL TRANSACTION, ALL OPERATIONS OF THE TRANSACTION ARE EXECUTED WITHIN THE SAME DATABASE.

EXAMPLE: FUNDS ARE TRANSFERRED FROM ONE ACCOUNT TO ANOTHER ACCOUNT WHERE BOTH ACCOUNTS BELONG TO THE SAME BANK.

2. GLOBAL TRANSACTION

IN A GLOBAL TRANSACTION, THE OPERATIONS OF A TRANSACTION ARE EXECUTED ACROSS DIFFERENT DATABASES.
EXAMPLE: FUNDS ARE TRANSFERRED FROM ONE ACCOUNT TO ANOTHER ACCOUNT WHERE BOTH ACCOUNTS BELONG TO DIFFERENT BANKS.

ACID PROPERTIES OF A TRANSACTION

ACID STANDS FOR:

A – ATOMICITY

ATOMICITY ENSURES THAT A TRANSACTION IS EXECUTED COMPLETELY OR NOT EXECUTED AT ALL. IF ANY OPERATION FAILS, THE ENTIRE TRANSACTION IS ROLLED BACK.
EXAMPLE: DURING A FUND TRANSFER, IF THE AMOUNT IS DEBITED FROM THE SENDER'S ACCOUNT BUT THE CREDIT TO THE RECEIVER'S ACCOUNT FAILS, THE DEBIT OPERATION IS ALSO CANCELED.

C – CONSISTENCY

CONSISTENCY ENSURES THAT A TRANSACTION BRINGS THE DATABASE FROM ONE VALID STATE TO ANOTHER VALID STATE WITHOUT VIOLATING ANY RULES OR CONSTRAINTS.
EXAMPLE: THE TOTAL BALANCE OF BOTH ACCOUNTS REMAINS CORRECT BEFORE AND AFTER A FUND TRANSFER.

I – ISOLATION

ISOLATION ENSURES THAT MULTIPLE TRANSACTIONS EXECUTING AT THE SAME TIME DO NOT INTERFERE WITH EACH OTHER.
EXAMPLE: IF TWO USERS ARE TRANSFERRING MONEY SIMULTANEOUSLY, EACH TRANSACTION IS EXECUTED INDEPENDENTLY.

D – DURABILITY

DURABILITY ENSURES THAT ONCE A TRANSACTION IS COMMITTED, ITS CHANGES ARE PERMANENTLY SAVED IN THE DATABASE, EVEN IF THERE IS A SYSTEM FAILURE OR POWER OUTAGE.
EXAMPLE: AFTER A FUND TRANSFER IS COMMITTED, THE UPDATED ACCOUNT BALANCES REMAIN SAVED EVEN IF THE DATABASE SERVER RESTARTS.

WHAT IS LOCKING?

LOCKING IS A MECHANISM USED BY THE DATABASE TO CONTROL CONCURRENT ACCESS TO DATA.
IT ENSURES THAT WHEN MULTIPLE USERS ACCESS THE SAME DATA SIMULTANEOUSLY, THE DATABASE MAINTAINS CONSISTENCY, INTEGRITY, AND PREVENTS CONFLICTS.

SHARED LOCK (READ LOCK)

A SHARED LOCK ALLOWS MULTIPLE USERS TO READ THE SAME DATA AT THE SAME TIME.
HOWEVER, NO USER CAN MODIFY (UPDATE OR DELETE) THE DATA UNTIL THE SHARED LOCK IS RELEASED.

CHARACTERISTICS

READ ONLY 
MULTIPLE USERS CAN READ SIMULTANEOUSLY
BLOCKS UPDATE AND DELETES


EXCLUSIVE LOCK

AN EXCLUSIVE LOCK ALLOWS A TRACTION TO MODIFY DATA

WHILE AN EXCLUSIVE LOCK EXITS:
NO OTHER TRACTION CAN READ USING A LOCKING READ OR MODIFY THAT ROW 
ONLY THE TRASACTION HOLDING THE LOCK CAN UPDATE/DELETE THE DATA

CHARACTERSITICS

USED FOR INSERT, UPDATE , DELETE
ONLY ONE EXCLUSIVE LOCK AT A TIME
BLOCKS BOTH READS (LOCKING READS) AND WRITES


INTENT LOCK

AN INTENT LOCK IS A TYPE OF LOCK THAT INDICATES A TRANSACTION INTENDS TO PLACE A SHARED OR EXCLUSIVE LOCK ON SPECIFIC ROWS OF A TABLE.
IT HELPS THE DATABASE MANAGE MULTIPLE TRANSACTIONS EFFICIENTLY AND PREVENTS LOCK CONFLICTS.


EXAMPLE FOR WRITE

LOCK TABLES EMPLOYEE WRITE;

UPDATE EMPLOYEE
SET SALARY = 90000.00
WHERE EMP_ID = 333;

UNLOCK TABLES;


Explanation
LOCK TABLES EMPLOYEE WRITE; → Locks the EMPLOYEE table for writing.
UPDATE EMPLOYEE SET SALARY = 90000.00 WHERE EMP_ID = 333; → Updates the salary of employee 333.
UNLOCK TABLES; → Releases the lock so other users can access the table.

Explanation
